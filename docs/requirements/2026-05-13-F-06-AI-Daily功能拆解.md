## 2. 多源自动抓取

### 触发条件

- 每日 8:00（北京时间）定时任务触发
- 也可手动触发（调试/补抓）

### 输入

| 数据项 | 说明 |
|--------|------|
| 信息源白名单 | 15-20 个源，含 URL、抓取方式（RSS/页面抓取/API）、所属分类 |
| 时间窗口 | 昨日 8:00 — 今日 8:00 |
| 关键词过滤 | AI、LLM、大模型、Agent、RAG、人工智能 等（中文）+ Artificial Intelligence, LLM, GPT, Claude 等（英文） |

### 处理流程

```
定时触发（8:00）
    │
    ▼
遍历信息源列表 ──────── 15-20 个源并行抓取
    │                       │
    │                       ├─ RSS 源 → 解析 RSS XML
    │                       ├─ 网页源 → HTTP GET + HTML 解析
    │                       └─ API 源 → HTTP GET JSON
    │
    ▼
关键词匹配过滤 ───────── 过滤非 AI 相关内容
    │
    ▼
时间窗口过滤 ─────────── 只保留发布时间在昨日 8:00 - 今日 8:00
    │
    ▼
原始条目存储 ─────────── 存入当日原始池，每条标记来源 + 原始标题 + 原始链接 + 发布时间
```

### 输出

- 原始条目池（50-150 条），每条包含：来源名称、原始标题、原始链接、发布时间、抓取时间戳

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 单个源无响应/解析失败 | 静默跳过，记录日志，继续下一个源 |
| 单个源超时（> 10s） | 重试 3 次，每次间隔 3s；全失败则跳过且标记"该源不可用" |
| 所有源不可用 | 触发兜底：页面展示"今日信息源暂不可用" + 手动重试按钮 |
| 某源返回非 AI 相关内容 | 关键词过滤自动剔除 |
| 时间窗口内无新内容（某源当日无更新） | 正常，该源返回 0 条 |

### 重试策略

```
第 1 次请求失败 → 等待 3s → 第 2 次请求
    → 失败 → 等待 3s → 第 3 次请求
        → 失败 → 跳过该源，标记不可用，log 记录
```

### 验收检查

- [ ] 8:00 准时触发
- [ ] 15-20 个源均被遍历，无遗漏
- [ ] 单源失败不影响其他源
- [ ] 时间窗口过滤准确，不包含窗口外内容
- [ ] 关键词过滤不误杀 AI 资讯（漏报率 < 5%）
- [ ] 全失败场景触发兜底态

## 3. 去重 + AI 筛选排序

### 触发条件

- 多源自动抓取完成后，原始条目池有数据时自动触发
- 原始条目池为空则不触发

### 输入

| 数据项 | 说明 |
|--------|------|
| 原始条目池 | 50-150 条，含来源/标题/链接/发布时间 |

### 处理流程

```
原始条目池（50-150 条）
    │
    ▼
  Step 1. AI 识别重复 ───── 标题相似度 + 语义相似度判断
    │                      │
    │                      判断标准：报道同一事件 = 重复
    │                      相似度阈值：标题相似 > 80% 或 语义相似 > 85%
    │
    ▼
  Step 2. 合并去重 ──────── 重复条目合并为一条
    │                      │
    │                      主展示来源选取：信息完整度评分（完整性 + 权威度加权）
    │                      副来源列表：所有报道此事件的来源链接
    │                      输出格式：主展示条目 + [多来源报道: A来源, B来源, C来源]
    │
    ▼
  去重后池（20-50 条）
    │
    ▼
  Step 3. AI 重要性评分 ──── 每条打分（1-10）
    │                      │
    │                      重要性权重：
    │                      - 新模型/产品发布：9-10
    │                      - 大厂战略/合作/并购：8-9
    │                      - 重要研究突破：7-10
    │                      - 工具/应用推荐：5-7
    │                      - 一般动态/观点/报告：3-6
    │
    ▼
  Step 4. 混合排序 ──────── 综合评分 = 重要性 × 0.7 + 时效性 × 0.3
    │                      │
    │                      时效性：越新越高（发布时间距现在越近）
    │
    ▼
  Step 5. 截取精选 ──────── Top 10-15 条进入日报
    │                      │
    │                      保证三类内容各至少 1 条
    │                      不足 10 条时全部展示，不填充
    │
    ▼
  精选条目池（10-15 条）
```

### 输出

- 精选条目池（10-15 条），每条含：主标题、合并后的来源列表、发布时间、去重标记（是否多源合并）、重要性评分

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 去重后的条目不足 10 条 | 全部展示，不填充；日报顶部标注"今日资讯较少" |
| 某类内容缺失（如今天无新模型发布） | 其他两类正常展示，不强行生成 |
| 重要性评分接近（多条同一分数区间） | 按时效性降序排列 |
| 所有条目均不通过关键词/质量筛选 | 日报展示兜底态"今日暂无高质量 AI 资讯" |

### 验收检查

- [ ] 同事件多源报道被正确合并（合并率 > 80%）
- [ ] 去重不误杀——不同事件不被合并（误杀率 < 5%）
- [ ] 每条日报 10-15 条，不超出范围
- [ ] 重要性排序符合直觉（大模型发布排在工具推荐前）
- [ ] 多来源标注可见且可点击

## 4. AI 分类打标

### 触发条件

- AI 筛选排序完成后，精选条目池确定后触发

### 输入

| 数据项 | 说明 |
|--------|------|
| 精选条目池 | 10-15 条，含标题、原文内容、来源 |

### 处理流程

```
精选条目池（10-15 条）
    │
    ▼
Step 1. AI 语义分类 ────── 输入标题 + 原文内容
    │                      │
    │                      三个分类：
    │                      - 新模型/新产品发布（模型/API/feature/版本更新）
    │                      - 大厂 AI 战略（投融资/组织变动/合作/政策/商业化）
    │                      - AI 工具/应用推荐（可直接使用的工具、插件、应用）
    │
    ▼
Step 2. 多标签判断 ─────── 跨类条目处理
    │                      │
    │                      示例：OpenAI 发布新 Agent 工具
    │                      → 主标签："新模型/新产品发布"（优先事件本身）
    │                      → 副标签："AI 工具/应用推荐"
    │
    ▼
Step 3. 置信度检查 ─────── 若 AI 分类置信度 < 0.7
    │                      归入"其他"，不强制分类
    │
    ▼
输出分类结果
```

### 输出

- 每条精选条目标注：主分类标签、副分类标签（可选，跨类时有）、置信度

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| AI 无法确定分类 | 置信度 < 0.7 时归入"其他"，不展示分类标签 |
| 分类结果严重失衡（如今天全是大厂动态） | 正常展示，不强制均衡 |
| 跨类条目 | 主标签 + 副标签，前端展示主标签颜色，副标签灰底显示 |

### 验收检查

- [ ] 分类准确率 > 90%（人工抽检 20 条为基准）
- [ ] 跨类条目正确标注主副标签
- [ ] 低置信度条目不误标分类
- [ ] 每条都有至少一个标签

## 5. AI 多级摘要

### 触发条件

- 分类打标完成后触发

### 输入

| 数据项 | 说明 |
|--------|------|
| 精选条目池 | 10-15 条，含标题、原文内容、分类标签 |

### 处理流程

```
精选条目池（10-15 条）
    │
    ▼
Step 1. 语言检测 ───────── 识别原文语言（中文/英文/其他）
    │
    ▼
Step 2. AI 翻译（英文 → 中文）── 仅英文新闻触发
    │                      │
    │                      翻译要求：保留专业术语原名
    │                      输出：中文标题 + 中文概要
    │                      保留英文原标题展示
    │
    ▼
Step 3. AI 生成概要 ────── 严格基于原文，不添加原文外信息
    │                      │
    │                      中文字数：200-300 字
    │                      结构：发生了什么 → 为什么重要 → 关键细节
    │
    ▼
Step 4. 事实锚定检查 ───── 每条概要可追溯到原文具体段落
    │                      │
    │                      不能凭空生成数据、评论、延伸观点
    │
    ▼
输出摘要池
```

### 输出

| 数据项 | 说明 |
|--------|------|
| 中文标题 | 中文新闻直接用原标题，英文新闻 AI 翻译 |
| 英文原标题 | 英文源新闻保留 |
| 概要 | 200-300 字 AI 中文概要 |
| 来源链接 | 主来源 + 多来源列表 |

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 原文内容不足以支撑 200 字概要 | 缩短至可用长度，不填充 |
| AI 翻译质量不佳（专有名词翻译错误） | 保留英文专有名词原文 |
| 摘要生成完全失败 | 保留原标题 + 来源链接，展示"原文直达" |

### 验收检查

- [ ] 每条摘要 200-300 字中文
- [ ] 英文新闻翻译后术语正确（重点检查：模型名、公司名、技术缩写）
- [ ] 摘要内容可追溯到原文，不添加原文外信息
- [ ] 英文新闻保留英文原标题

## 6. 来源链接标注

### 触发条件

- 日报生成全流程必须携带来源，非独立步骤

### 输入

| 数据项 | 说明 |
|--------|------|
| 精选条目池 | 含主来源 + 多来源列表 |

### 处理流程

```
每条精选条目
    │
    ▼
标注主来源链接 ──────────── 信息完整度评分最优的来源
    │
    ▼
检查多来源标记 ─────────── 若为合并条目，列出所有来源链接
    │                      │
    │                      展示格式：[A来源] 首发 · [B来源] 详细报道 · [C来源] 另一视角
    │
    ▼
链接有效性校验 ─────────── 链接必须可点开，不展示死链
```

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 原始链接失效（已删除/不可访问） | 保留链接但标记"链接可能已失效" |
| 单来源条目 | 仅展示一个来源链接，不标注多来源 |

### 验收检查

- [ ] 100% 条目附带来源链接
- [ ] 链接格式正确，可直接点击打开
- [ ] 多来源合并条目展示了全部来源

## 7. 大模型趋势看板

### 7.1 模型性能排行

#### 触发条件

- 每日与日报同步更新，8:00 触发

#### 输入

| 数据项 | 说明 |
|--------|------|
| artificialanalysis.ai API | `/api/v2/data/llms/models`，含 evaluations（评分）、pricing（价格）、speed（速度）|

#### 处理流程

```
API 拉取数据（8:00）
    │
    ▼
提取 Top 10 ────────────── 按 artificial_analysis_intelligence_index 降序排列
    │
    ▼
提取分项评分 ───────────── 每模型展示：
    │                      ├─ 综合评分（intelligence_index）
    │                      ├─ 编码（coding_index）
    │                      ├─ 数学（math_index）
    │                      └─ 推理（mmlu_pro / gpqa）
    │
    ▼
渲染输出 ──────────────── 表格（详细数据）+ 横向柱状图（综合评分排名）
```

#### 输出

- Top 10 模型表格 + 综合评分柱状图

#### 异常处理

| 场景 | 处理方式 |
|------|---------|
| API 返回数据中某模型缺少某项评分 | 表格该单元格显示"-" |
| API 不可用 | 展示上次缓存数据 + 标注"数据更新于 X 日 X 时" |

#### 验收检查

- [ ] Top 10 排名与 artificialanalysis.ai 网站一致
- [ ] 表格 + 柱状图数据一致
- [ ] API 异常时展示缓存数据

### 7.2 价格趋势

#### 输出

- 模型价格对比表，按混合价格升序排列

#### 验收检查

- [ ] 价格数据与 API 返回值一致
- [ ] 价格单位为 $/每百万 token，标注清楚

- 输出速度对比图 + 首 token 延迟对比图

## 9. 微信推送通知

### 触发条件

- 日报页面生成完成后触发（预计 8:05-8:10）

### 输入

| 数据项 | 说明 |
|--------|------|
| 日报页面 URL | 当日日报静态页面地址 |
| Webhook 配置 | 企业微信机器人 Webhook URL |

### 处理流程

```
日报页面部署完成
    │
    ▼
构建推送消息 ────────────
    │                     消息格式：
    │                     标题：AI Daily | YYYY-MM-DD
    │                     内容：今日要点一句话总结（50 字内）
    │                     链接：点击查看完整日报
    │
    ▼
发送 Webhook 请求 ─────── POST 企业微信机器人 Webhook
    │
    ▼
发送结果处理
```

### 输出

- 微信推送成功/失败状态

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| Webhook 请求失败 | 重试 1 次，仍失败则记录日志，不阻塞日报页面可正常访问 |
| 消息内容超过企业微信限制 | 截断概要至 200 字符 |
| Webhook URL 失效 | log 报错，页面正常生成 |

### 验收检查

- [ ] 日报生成完成后自动推送
- [ ] 推送消息包含日期、要点总结、查看链接
- [ ] 推送失败不影响页面正常访问

## 10. 页面阅读

### 触发条件

- 用户点击微信推送链接或直接访问日报 URL

### 输入

## 11. 分级展开

### 触发条件

- 用户浏览日报条目时点击操作

### 交互流程

```
用户看到条目列表（默认态）
    │  每条展示：类型标签 + 标题 + 来源 + 时间
    │
    ▼
点击条目 ──────────────── 展开 AI 概要（200-300 字）
    │                      标题保持可见，概要区平滑展开
    │
    ▼
点击概要区【查看原文】 ──── 新窗口打开原始来源链接
    │
    ▼
点击其他位置 / 再次点击 ── 收起概要
```

### 交互细节

| 层级 | 展示内容 | 触发方式 |
|------|---------|---------|
| L1 标题层 | 类型标签 + 标题 + 来源 + 时间 | 默认可见 |
| L2 概要层 | 200-300 字 AI 概要 + "查看原文" 按钮 | 点击展开 |
| L3 原文层 | 跳转原始来源网站 | 点击"查看原文" |

### 功能限制

- 同时只展开一条概要（展开新条目时自动收起旧条目）
- 展开状态不持久化（刷新页面回到 L1）

### 验收检查

- [ ] 点击展开动画流畅
- [ ] 同时间只展开一条
- [ ] 原文链接可跳转

## 14. 定时触发

### 触发条件

- 每日 8:00（北京时间）自动执行，无需人工干预
- 支持手动触发（调试/补抓）

### 执行流程

```
8:00 定时触发
    │
    ├── Step 1. 多源抓取（并行，约 30s-60s）
    ├── Step 2. 去重 + AI 筛选排序（约 10s-20s）
    ├── Step 3. AI 分类打标（约 10s）
    ├── Step 4. AI 多级摘要（约 30s-60s）
    ├── Step 5. 大模型趋势数据拉取（约 5s-10s）
    ├── Step 6. 静态页面生成 + 部署（约 10s-30s）
    └── Step 7. 微信推送（约 2s）
    │
    ▼
预计 8:02-8:10 完成全流程
```

### 实现方式

- GitHub Actions scheduled workflow 或 crontab
- 日志记录每次执行状态（成功/失败/耗时）

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 某步骤执行超时 | 超时 60s 后跳过该步骤，继续后续流程 |
| 全流程失败 | 记录错误日志，保留昨日页面不覆盖 |
| 定时任务漏触发 | 支持手动触发补充执行 |

### 验收检查

- [ ] 每日 8:00 准时触发
- [ ] 全流程自动化无人工干预
- [ ] 执行日志可追溯
- [ ] 手动触发可用

## 15. 信息源管理

### v1.0 方案：配置文件

```
sources.json
[
  {
    "name": "量子位",
    "url": "https://www.qbitai.com/feed",
    "type": "rss",
    "language": "zh",
    "category": "中文媒体",
    "enabled": true
  },
  {
    "name": "TechCrunch AI",
    "url": "https://techcrunch.com/category/artificial-intelligence/",
    "type": "web",
    "language": "en",
    "category": "英文媒体",
    "enabled": true
  }
  // ... 15-20 个源
]
```

### 配置字段

| 字段 | 说明 |
|------|------|
| name | 信息源名称 |
| url | 抓取地址（RSS/页面/API） |
| type | 抓取方式：rss / web / api |
| language | 语言：zh / en |
| category | 所属分类 |
| enabled | 是否启用（临时关闭源时不删除配置） |

### v2.0 规划：管理页面

- 简单的 Web 管理页，增删改查信息源
- 开启/关闭源（不删除历史数据）
- 抓取日志查看

### 验收检查

- [ ] 配置文件格式正确
- [ ] 新增/禁用源只需编辑配置文件
- [ ] 配置文件被误删时有备份/重新生成能力
